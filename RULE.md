# 本项目协作规则（用户需求汇总）

> 目的：记录你到目前为止对“我如何回答/如何改项目”的要求，后续协作以此为准。

## 1) 回复与沟通方式

- **默认用中文回复**（除非你明确要求英文）。
- 你贴出报错/日志时：先用中文说明“发生了什么 + 可能原因 + 合适的解决方案/排查路径”。
- 当你明确说 **“先不要修改代码/不修改代码”**：只做分析、给方案或指导，不改代码。

## 2) 代码修改的工作流（强制）

- **每次要改功能/修 bug：先输出详细修改方案**（做什么、改哪些文件、关键逻辑怎么实现、验证方式），并等待你确认后再动代码。
- 在满足需求前提下：**尽量少改动、避免无关重构**（“能跑且可维护”优先）。
- 任何改动完成后：总结变更点与验证方式（便于你复查）。

## 3) 变更记录与文档同步（强制）

- **之后每次修改都必须记录**到 `docs/CHANGELOG.md`（写“改了什么/为什么/涉及哪些文件/怎么验证”）。
- 变更会影响项目说明时：同步更新 `docs/PROJECT_OVERVIEW.md`（保持文档与现状一致）。

## 4) 密钥/隐私与配置（强制）

- **任何文档/代码仓库内禁止写入真实 API Key/Token/账号密码**。
- `.env.example` 只放**变量名与示例**；真实密钥只写入本机 `.env`（被 gitignore 忽略）。
- 在 `docs/CHANGELOG.md` 里只能写“已新增/已配置某环境变量”，不能写具体值。

## 5) 产品与交互层面的约束（持续遵守）

- 全站 UI 文案：**中文显示**；练习题题干：**保持英文**。
- 练习题生成：
  - 题型为**可勾选项**（默认全选）；题目数量可调（默认 10）。
  - 不按“每个单词每种题型各一题”的固定模式；而是**随机混合不同题型**（在勾选范围内）。
  - 选择题风格需更贴近**国内英语试卷**（如单句单空填空、A/B/C/D 选项、含词形变化等）。
- 选词用于生成（练习/故事）：
  - 必须保留“最近一周/最近一个月”等时间范围选项（按你的最新决定可为跨分组口径）。
  - 当选择单词过多：需要提示并让你选择是否继续（避免超时/失败）。
- 故事功能：
  - 生成后**直接在界面展示**；
  - 提供“导出”按钮，当前先支持**导出 PDF**。
- 单词本整体体验：改动需符合“英语学习软件”定位；**单词本的基础形态尽量保持稳定**（在不破坏原有逻辑下扩展能力）。
